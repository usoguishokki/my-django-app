# Frontend Architecture Guide (SOLID / DRY)

This document defines a **reusable, scalable frontend architecture** for building features using:

* HTML
* JavaScript (ES Modules)
* SCSS
* Django (Template + API)

The goal is to ensure:

* Maintainability
* Scalability
* Readability
* Reusability

---

# 1. Core Principles

## 1.1 SOLID

| Principle                 | Meaning                                |
| ------------------------- | -------------------------------------- |
| S (Single Responsibility) | One file = one responsibility          |
| O (Open/Closed)           | Extend without modifying existing code |
| L (Liskov Substitution)   | Replaceable components                 |
| I (Interface Segregation) | Avoid unnecessary dependencies         |
| D (Dependency Inversion)  | Depend on abstractions, not details    |

---

## 1.2 DRY

* Avoid duplicated logic
* Centralize reusable functions
* Separate responsibilities clearly

---

# 2. Standard Folder Structure

Each feature should follow this structure:

```
static/js/
└─ <feature>/
   ├─ domain/
   ├─ ui/
   ├─ application/
   └─ pages/
```

---

# 3. Layer Definitions

---

## 3.1 domain (Business Logic Layer)

### Responsibility

* Pure logic
* Data transformation
* State management

### Rules

* No DOM access
* No HTML generation
* No API calls

### Examples

```javascript
// CalendarDateService.js
export class CalendarDateService {
  static createMonthMatrix(year, month) {}
}
```

```javascript
// CalendarState.js
export class CalendarState {
  moveMonth(delta) {}
}
```

---

## 3.2 ui (Presentation Layer)

### Responsibility

* Rendering
* DOM manipulation
* HTML generation

### Rules

* No business logic
* No state mutation

### Internal Structure

```
ui/
├─ elements.js   // DOM selectors
├─ Template.js   // HTML generation
└─ Renderer.js   // DOM rendering
```

---

### Examples

```javascript
// elements.js
export function getElements(root) {
  return {
    container: root.querySelector('[data-role="container"]'),
  };
}
```

```javascript
// Template.js
export class Template {
  static create(viewModel) {}
}
```

```javascript
// Renderer.js
export class Renderer {
  render(root, viewModel) {
    root.innerHTML = Template.create(viewModel);
  }
}
```

---

## 3.3 application (Controller Layer)

### Responsibility

* Event handling
* State updates
* Coordination between domain and ui

### Rules

* No direct HTML generation
* No business logic implementation

---

### Example

```javascript
export class Controller {
  constructor({ state, renderer, elements }) {
    this.state = state;
    this.renderer = renderer;
    this.elements = elements;
  }

  init() {
    this.bindEvents();
    this.render();
  }

  bindEvents() {}

  render() {}
}
```

---

## 3.4 pages (Entry Point Layer)

### Responsibility

* Dependency injection
* Initialization only

### Rules

* No logic
* No rendering
* No state manipulation

---

### Example

```javascript
document.addEventListener('DOMContentLoaded', () => {
  const controller = new Controller({...});
  controller.init();
});
```

# 4. Dependency Rules

## 4.1 Allowed Direction

```
pages
  ↓
application
  ↓
domain
  ↓
ui (Template only)
```

---

## 4.2 Forbidden Patterns

### ❌ domain accessing DOM

```javascript
document.querySelector(...)
```

---

### ❌ Controller doing everything

```javascript
// NG
fetch()
buildHTML()
updateState()
```

---

### ❌ Renderer containing business logic

```javascript
if (isBusinessRule) { ... }
```

---

# 5. ViewModel Pattern

## 5.1 Definition

UI should only consume **ViewModel**, not raw data.

---

## 5.2 Example

```javascript
{
  year: 2026,
  month: 3,
  weeks: [
    [
      {
        day: 20,
        isoDate: "2026-03-20",
        isToday: true,
        isCurrentMonth: true,
        events: []
      }
    ]
  ]
}
```

---

## 5.3 Rules

* domain generates ViewModel
* ui consumes ViewModel only
* no logic in Template

# 7. Implementation Rules

---

## 7.1 One Responsibility per File

### ❌ Bad

```javascript
// everything in one file
```

### ✅ Good

```javascript
CalendarService
CalendarRenderer
CalendarController
```

---

## 7.2 Pure Functions in domain

### ✅ Good

```javascript
input → output
```

### ❌ Bad

```javascript
side effects
```

---

## 7.3 DOM Access Only in UI

---

## 7.4 State Centralization

### ❌ Bad

```javascript
this.currentMonth = ...
```

### ✅ Good

```javascript
state.moveMonth()
```

3. Connect via Controller
4. Add API integration
5. Extend UI

* [ ] Easy to extend
