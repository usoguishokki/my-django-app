# 開発・本番運用ルール（完成版）

## 1. 概要
本プロジェクトでは、開発環境と本番環境を分離し、安全に機能追加・修正を行う。  
現在の運用は **`settings.py` 一本運用** とし、開発環境・本番環境ともに `myproject.settings` を使用する。

---

## 2. 環境構成

### 開発環境
- パス: `E:\repos\myproject`
- 仮想環境: `venv`
- 設定: `myproject.settings`
- 起動: VSCodeデバッグ or `runserver`

### 本番環境
- パス: `C:\inetpub\wwwroot\sitefolder\myproject`
- 仮想環境: `django_iis_env`
- 設定: `myproject.settings`
- 起動: IIS (wfastcgi / wsgi.py)

---

## 3. settings運用ルール

### 現在の方針
- `settings.py` 一本で運用する
- `manage.py` / `wsgi.py` / `web.config` はすべて `myproject.settings` を参照する
- 本番環境には **`myproject/settings/` ディレクトリを配置しない**

### 理由
`settings.py` と `settings/` ディレクトリが共存すると、`myproject.settings` の import 解決が衝突し、IIS で 500 エラーになることがあるため。

### 禁止事項
- `myproject/settings.py` と `myproject/settings/` を共存させる
- settings 構成変更をしたのに `web.config` / `wsgi.py` / `manage.py` の参照先を揃えない

---

## 4. 起動方法

### 開発
```python
# manage.py
os.environ.setdefault('DJANGO_SETTINGS_MODULE', 'myproject.settings')
```

```powershell
python manage.py runserver
```

### 本番（IIS）
```python
# myproject/wsgi.py
os.environ.setdefault('DJANGO_SETTINGS_MODULE', 'myproject.settings')
```

```xml
<!-- web.config -->
<environmentVariable name="DJANGO_SETTINGS_MODULE" value="myproject.settings" />
<environmentVariable name="PYTHONPATH" value="C:\inetpub\wwwroot\sitefolder\myproject" />
<environmentVariable name="WSGI_HANDLER" value="django.core.wsgi.get_wsgi_application()" />
```

## 5. Git運用

### 開発の流れ
```text
開発 → 動作確認 → commit → push
```

### 本番反映
```text
本番環境で pull
```

### 注意
- 本番環境で直接コード修正しない
- settings 構成を変更した場合は、先に開発環境で動作確認する
- 本番環境での手修正は障害復旧時のみ。後で必ず開発環境へ反映する

## 7. staticfiles運用

### settings
```python
STATIC_URL = '/static/'
STATICFILES_DIRS = [
    BASE_DIR / 'myapp' / 'static',
]
STATIC_ROOT = BASE_DIR / 'staticfiles'
```

### ルール
- `staticfiles` はGit管理しない
- 本番で `collectstatic` を実行する
- CSS / JS / 画像 / SCSS に変更がある場合は、本番反映時に必ず `collectstatic` を実行する
- Pythonコードのみ変更でも、迷ったら `collectstatic --noinput` まで実行する

```text
① 開発環境で修正

④ git push

⑥ python manage.py check
⑦ python manage.py migrate
⑧ python manage.py collectstatic --noinput
⑨ iisreset

```

## 9-1. 本番反映前の事前確認

```powershell
# 現在のコミットIDを控える
git rev-parse HEAD

# 作業ツリー確認
git status

# ブランチ確認
git branch --show-current

# リモート確認
git remote -v
```

### 必須確認
- コミットIDをメモ
- DBバックアップ取得
- migration有無確認
- `web.config` / `manage.py` / `wsgi.py` の settings 参照先が `myproject.settings` で揃っているか確認

## 9-2. 本番反映手順

```powershell

# 設定確認

# DB反映
python manage.py migrate

# 静的ファイル反映
python manage.py collectstatic --noinput

# IIS再起動
iisreset
```

- ログイン確認
- 対象画面確認
- CSS崩れなし
- JSエラーなし
- エラーログなし

## 9-4. ロールバック手順

### コードのみ戻す
```powershell
git reset --hard <元のコミットID>
python manage.py check
python manage.py collectstatic --noinput
iisreset
```

### DB含む場合
- 事前バックアップから復元
- または `migrate` 巻き戻し

## 9-5. 本番運用ルール

### 必須
```powershell
python manage.py check
python manage.py migrate
python manage.py collectstatic --noinput
```

### 禁止
```text
- settings 構成変更を未確認のまま本番へ入れる
- 本番環境に settings ディレクトリを残したまま settings.py 運用する
- バックアップ未確認で migrate する
```

## 10. 500エラー発生時の確認手順

500エラーが出た場合は、推測で直さずに以下の順で確認する。

### 10-1. まず Django 単体で動くか確認
```powershell
python manage.py check
python manage.py runserver 127.0.0.1:8080 --insecure
```

#### 確認ポイント
- `check` が通るか
- `runserver --insecure` で画面が表示できるか

#### 判断
- Django単体で動く → IIS / FastCGI / web.config 側の問題
- Django単体でも落ちる → Django設定やコード側の問題

---

### 10-2. IISサイト情報を確認
```powershell
& "$env:windir\System32\inetsrv\appcmd.exe" list site
```

#### 確認ポイント
- 対象サイト名
- Site ID
- バインドポート

---

### 10-3. Failed Request Tracing を有効化
```powershell
& "$env:windir\System32\inetsrv\appcmd.exe" set site "DjangoApp" /traceFailedRequestsLogging.enabled:true /traceFailedRequestsLogging.directory:"C:\inetpub\logs\FailedReqLogFiles"
& "$env:windir\System32\inetsrv\appcmd.exe" set config "DjangoApp" -section:system.webServer/tracing/traceFailedRequests /+"[path='*']"
```

その後、IIS Manager または appcmd で以下の内容になるようにする。

```xml
<add path="*">
  <traceAreas>
    <add provider="WWW Server" areas="Authentication,Security,Filter,StaticFile,CGI,Compression,Cache,RequestNotifications,Module,FastCGI" verbosity="Verbose" />
  </traceAreas>
  <failureDefinitions statusCodes="500-599" />
</add>
```

---

### 10-4. 500エラーを再現してトレースを見る
```powershell
Get-ChildItem C:\inetpub\logs\FailedReqLogFiles\W3SVC1 -Filter *.xml |
Sort-Object LastWriteTime -Descending |
Select-Object -First 5 LastWriteTime, FullName
```

最新の XML を確認する。

```powershell
Get-Content C:\inetpub\logs\FailedReqLogFiles\W3SVC1\fr000002.xml -Tail 200
```

必要に応じて FastCGI 周辺だけを見る。

```powershell
Select-String -Path C:\inetpub\logs\FailedReqLogFiles\W3SVC1\fr000002.xml -Pattern "MODULE_SET_RESPONSE_ERROR_STATUS|GENERAL_SET_RESPONSE_ERROR_STATUS|ErrorCode|Notification|HttpStatus|HttpReason|ModuleName|HandlerName|FastCgiModule|CGI|Python|wfastcgi" -Context 3,3
```

---

### 10-5. 今回実際に発生した代表例
#### エラー内容
```text
AttributeError: 'Settings' object has no attribute 'ROOT_URLCONF'
```

#### 原因
- `myproject.settings` を参照しているのに
- `myproject/settings.py` と `myproject/settings/` ディレクトリが衝突した

#### 対処
- `settings.py` 一本運用に戻す
- 本番環境から `myproject/settings/` ディレクトリを削除する
- `manage.py` / `wsgi.py` / `web.config` を `myproject.settings` に統一する

---

### 10-6. IIS設定の確認ポイント
`web.config` の environmentVariables を確認する。

```xml
<environmentVariables>
  <environmentVariable name="DJANGO_SETTINGS_MODULE" value="myproject.settings" />
  <environmentVariable name="PYTHONPATH" value="C:\inetpub\wwwroot\sitefolder\myproject" />
  <environmentVariable name="WSGI_HANDLER" value="django.core.wsgi.get_wsgi_application()" />
</environmentVariables>
```

#### 確認ポイント
- `DJANGO_SETTINGS_MODULE` が `myproject.settings`
- `PYTHONPATH` が本番プロジェクトパス
- `WSGI_HANDLER` が `django.core.wsgi.get_wsgi_application()`

---

### 10-7. 500エラー時の優先順位
```text
① python manage.py check
② python manage.py runserver 127.0.0.1:8080 --insecure
③ web.config の参照先確認
④ Failed Request Tracing で原因取得
⑤ FastCGI / wfastcgi / settings衝突を確認
```

---

## 11. まとめ

- 開発と本番は完全分離
- `settings.py` 一本運用
- Git経由でのみ反映
- staticfilesは本番で生成
- ロールバック可能な状態で作業
- 500エラー時は Django 単体確認 → IIS / FREB 解析の順で切り分ける
